## 3. 范围一致性检查

### C-S1：.NET 版本不一致 ⚠️ P0

| 文档 | 写法 |
|------|------|
| 01-立项单 §1 | "ASP.NET Core MVC (.NET **8**)" |
| 03-PRD-Lite §1 | "ASP.NET Core MVC (.NET **8**)" |
| 07-系统设计 §2 | ".NET **6.0**" |
| README 技术栈 | ".NET **6.0**" |
| 代码实际 | `TargetFramework=net6.0` |

**冲突**：01和03写 .NET 8，07和代码用 .NET 6。07还写"API 兼容 .NET 8"，但立项单和PRD未体现此兼容说明。

**影响**：P0。开发环境配置、依赖版本都依赖此口径。若按01/.NET 8执行，EF Core 包版本需升至8.x，与07/代码的6.0不兼容。

**建议回改**：统一为 .NET 6（与代码一致），或统一为 .NET 8（需全量升级）。

### C-S2：页面总数口径不一致 ⚠️ P1

| 文档 | 页面数 | 说明 |
|------|--------|------|
| 01-立项单 §7 | "约**10**个页面" | 含取消预约 |
| 02-MVP确认 §5 | "**10**个视图页面" | 含取消预约 |
| 03-PRD-Lite §7 | "**9**个视图页面" | 取消预约为POST，不计入 |
| 04-页面树 §1 | "**9**页" | 取消预约为POST，不计入 |
| README | "用户端5页 + 管理端4页 = 9页" | 同04 |

**冲突**：01/02写10页，03/04/README写9页。03已注明"两种口径均合理"，但01/02未同步更新。

**影响**：P1。不影响功能实现，但造成文档间引用混乱。

## 4. 页面一致性检查

### C-P1：管理端路由命名不一致 ⚠️ P0

| 页面 | 01-立项单 | 04-页面树 | 07-系统设计 | 代码实际 |
|------|----------|----------|------------|---------|
| 座位管理 | `/Admin/Seat/Index` | `/Admin/Seat/Index` | `/Admin/SeatIndex` | `/Admin/SeatIndex` |
| 预约管理 | `/Admin/Reservation/Index` | `/Admin/Reservation/Index` | `/Admin/ReservationIndex` | `/Admin/ReservationIndex` |
| 统计页 | `/Admin/Statistics/Index` | `/Admin/Statistics/Index` | `/Admin/Statistics` | `/Admin/Statistics` |

**冲突**：01和04写`/Seat/Index`、`/Reservation/Index`、`/Statistics/Index`，07和代码用`/SeatIndex`、`/ReservationIndex`、`/Statistics`。

**影响**：P0。路由是前后端对接的核心约定，不一致会导致页面跳转失败。

**建议回改**：统一以代码实现为准（`/SeatIndex`、`/ReservationIndex`、`/Statistics`），回改01和04中的路由。

### C-P2：页面卡布局要求与立项单范围冲突 ⚠️ P1

| 文档 | 内容 |
|------|------|
| 01-立项单 §6 | "移动端适配（本项目仅适配 **PC 端浏览器**）" — 列为暂不开发 |
| 05-页面卡 U1 §11 | "手机竖屏：垂直居中布局...按钮纵向排列" |
| 05-页面卡 U2 §11 | "手机竖屏：卡片式列表...底部按钮全宽" |
| 05-页面卡（全部） | 每张页面卡都有"手机竖屏"布局要求 |

**冲突**：01明确排除移动端适配，05为每个页面都定义了手机竖屏布局。

**影响**：P1。功能不受影响（手机端是额外工作），但05的手机端布局要求与01范围矛盾。

### C-P3：原型假数据座位数与种子数据不一致 ⚠️ P1

| 文档 | 座位数 |
|------|--------|
| 06-原型 §2.1 | 10条假数据（A-01~A-03, B-01~B-03, C-01~C-03, D-01） |
| 08-数据库设计 §4.1 | 20条种子数据（A区5+B区5+C区5+D区5） |
| 02-MVP确认 | 20条座位 |

**冲突**：原型用10条假数据，数据库设计和MVP确认用20条。

**影响**：P1。原型仅作展示不影响开发，但假数据量与实际差距较大。

## 5. 数据一致性检查

### C-D1：AdminUser 表缺少 CreatedAt 字段 ⚠️ P0

| 文档 | AdminUser 字段 |
|------|---------------|
| 03-PRD-Lite §11 | Id, Username, Password（3个字段） |
| 08-数据库设计 §4.3 | Id, Username, Password, CreatedAt（4个字段） |
| 代码实际 | Id, Username, Password, CreatedAt（4个字段） |

**冲突**：03-PRD-Lite只定义了3个字段，08和代码有4个字段（多了CreatedAt）。

**影响**：P0。PRD是需求基准，字段不一致会导致后续审计时无法追溯。

**建议回改**：在03-PRD-Lite §11 AdminUser表中补充 `CreatedAt` 字段。

### C-D2：统计缓存口径不一致 ⚠️ P1

| 文档 | 口径 |
|------|------|
| 03-PRD-Lite §12 | "统计数据实时更新，**无缓存**" |
| 08-数据库设计 §10 | "统计数据实时计算，**无缓存**" |
| 10-开发准备 §7.2 | "统计数据实时计算，**无缓存**" |
| 代码实际 | 使用 `IMemoryCache`，缓存5分钟 |

**冲突**：03/08/10均写"无缓存"，代码实际有5分钟缓存。

**影响**：P1。功能不受影响（缓存是优化），但文档与实现不一致。

## 12. 开发前 Go / No-Go 结论

### Go 条件评估

| 条件 | 状态 | 说明 |
|------|------|------|
| 核心范围无冲突 | ⚠️ | .NET版本和路由命名存在P0冲突，但不影响功能逻辑 |
| 页面定义无冲突 | ⚠️ | 路由命名P0冲突，需回改 |
| 数据模型无冲突 | ⚠️ | AdminUser缺字段P0冲突，需回改 |
| 链路设计无冲突 | ✅ | 7条链路完整，与页面/Controller对应 |
| 验收标准可追溯 | ✅ | 01/03成功标准可映射 |
| Sprint规划完整 | ✅ | 0-4 Sprint + 任务板 + 里程碑 |

### 结论

**⚠️ 有条件 Go**

3个P0阻塞项均为**文档间命名/字段不一致**，不涉及功能逻辑矛盾。代码实现已通过验证（81/81测试通过），说明功能本身是正确的。

**Go 的前提**：在进入 Sprint 1 编码前，完成以下3项P0回改：
1. 统一 .NET 版本为 6.0（回改01、03）
2. 统一管理端路由命名（回改01、04）
3. AdminUser 补充 CreatedAt 字段（回改03）

**不 Go 的风险**：若不回改，后续文档引用时会出现版本/路由/字段不一致，导致审计时反复修改。
