## 3. 代码审查发现

### D-01. SeatService 存在 fire-and-forget 问题（已评估-不修复）

**文件**：`Services/SeatService.cs:16-18`

**问题**：`CreateSeatAsync`、`UpdateSeatAsync`、`DeleteSeatAsync` 未 `await` 调用 Repository，异常被静默吞掉。

**评估**：AdminController 直接注入 `ISeatRepository`（不走 SeatService），当前所有座位操作已正确 `await`。此缺陷不影响现有功能。

**处置**：记录为已知限制，后续重构时修复。不阻塞当前交付。

---

### D-02. SwitchUser 无鉴权（设计如此-不修复）

**文件**：`Controllers/ReservationController.cs:111-116`

**问题**：`SwitchUser` 允许任意用户切换，无鉴权。

**评估**：本项目为演示系统，体验账号切换是设计功能（docs/05 §用户端要求"支持切换体验账号"）。非安全缺陷。

**处置**：无需修复。

---

### D-03. 预约日期缺少服务端过去日期校验（低风险-记录）

**文件**：`Controllers/ReservationController.cs:34`

**问题**：POST Create 未校验 `reserveDate >= DateTime.Today`。HTML `min` 属性仅做客户端限制。

**评估**：客户端 `min` 属性已阻止正常用户选择过去日期。绕过客户端直接提交过去日期的技术上可行，但后果仅是创建一条无效预约（不影响数据完整性）。低风险。

**处置**：记录为已知限制。如需修复，可在 POST Create 添加一行 `if (reserveDate.Date < DateTime.Today) return BadRequest();`。

---

### D-04. AdminController 手动 IsLoggedIn 检查（设计如此-不修复）

**文件**：`Controllers/AdminController.cs` 所有 Action

**问题**：每个 Action 手动调用 `IsLoggedIn()`，遗漏则暴露管理端。

**评估**：当前所有 Action 已正确添加检查。项目规模小（7个Action），手动检查可控。使用 `[Authorize]` 需要额外配置 Authentication 中间件，超出本项目范围。

**处置**：无需修复。

---

### D-05. 并发预约无锁（设计范围外-不修复）

**文件**：`Services/ReservationService.cs:21-28`

**问题**：`HasConflictAsync` 和 `AddAsync` 之间无事务锁，并发请求可能创建重复预约。

**评估**：演示系统，单用户并发场景极少。EF Core 默认隔离级别（Read Committed）在大多数情况下足够。

**处置**：记录为已知限制，后续如需生产部署再处理。
