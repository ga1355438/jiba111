### 3.2 管理端闭环走查

```
管理员登录 (/Admin/Login)
  └─→ POST 验证 → Session["AdminUsername"] = admin
        └─→ 座位管理 (/Admin/SeatIndex)
              ├─→ 新增座位 (/Admin/SeatCreate) → POST → 写入 Seat 表
              ├─→ 编辑座位 (/Admin/SeatEdit/{id}) → POST → 更新 Seat 表（不修改 CreatedAt）
              └─→ 删除座位 (POST /Admin/SeatDelete/{id}) → 删除 Seat 表记录
        └─→ 预约管理 (/Admin/ReservationIndex)
              └─→ 按日期/状态筛选 → 分页显示
        └─→ 统计 (/Admin/Statistics)
              └─→ 总预约数/今日预约/有效预约/热门TOP5/时段分布（缓存5分钟）
        └─→ 退出登录 (/Admin/Logout) → 清除 Session
```

### 3.3 权限控制验证

| 检查项 | 结果 | 说明 |
|--------|------|------|
| 管理端所有 Action 需登录 | ✅ | IsLoggedIn() 检查，未登录重定向到 Login |
| 用户端无需登录 | ✅ | 用户端所有页面无登录检查 |
| Session 隔离 | ✅ | AdminUsername（管理端）和 CurrentUserName（用户端）使用不同 Key |
| 密码安全 | ✅ | PasswordHasher 哈希存储，不明文保存 |
| 删除座位校验 | ✅ | 有预约记录时 EF Core 外键约束阻止删除 |

### 3.4 状态回流验证

| 管理端操作 | 数据库变化 | 用户端可见变化 |
|-----------|-----------|--------------|
| 编辑座位 Status=1（维护中） | Seat.Status = 1 | 座位列表显示"维护中"标签，详情页按钮禁用 |
| 编辑座位 Status=0（可用） | Seat.Status = 0 | 座位列表显示"可用"标签，详情页可预约 |
| 新增座位 | Seat 表新增记录 | 座位列表可见新座位 |
| 删除座位 | Seat 表删除记录 | 座位列表不再显示该座位 |

| 补足项 | 说明 | 原因 |
|--------|------|------|
| 管理端使用 Session 鉴权而非 Cookie/Token | 课堂项目简化实现 | docs/07 §9.1 明确使用 Session |
